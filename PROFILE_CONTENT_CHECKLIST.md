# 个人主页内容发布清单

这个仓库的 `README.md` 会在仓库名与用户名都为 `peteryipikachu-cpu` 时显示在 GitHub 个人主页。发布前，请完成下面的替换，避免把模板内容当作履历事实。

## 1. 首屏与联系入口

- 将昵称替换为希望招聘方使用的姓名或稳定网名。
- 若有公开简历，添加一个可长期访问的链接；若使用邮箱，优先单独的求职邮箱。
- 仅保留持续维护的平台链接，例如 LinkedIn、博客、知乎或 Bilibili。

## 2. 技术标签

保留实际做过项目的技术，不以“熟练度百分比”标注能力。每项技术最好能在一个公开仓库中找到证据。

可替换或删除：Python、Pandas、scikit-learn、PyTorch、Jupyter、FastAPI、Docker、SQL、Airflow、Spark。

## 3. 精选项目卡片

用真实项目替换 `README.md` 中的三行占位内容，并让每个项目 README 至少包括：

1. **问题与受众**：解决什么问题，使用者是谁。
2. **你的职责**：你具体完成了哪些工作。
3. **数据与许可**：数据来源、规模、预处理及使用限制。
4. **方法**：模型、特征、基线或系统架构。
5. **结果**：数据集、指标、对照条件与结果；没有可靠指标时写明已完成的能力，不写虚构数字。
6. **快速开始**：环境、安装、数据准备与最小运行命令。
7. **局限与下一步**：已知失败场景、偏差或待完成工作。

示例（仅在数据真实时使用）：

```md
| [评论情感分类](https://github.com/peteryipikachu-cpu/repo-name) | 负责数据清洗、基线与误差分析；在公开数据集上以 macro-F1 评估 | [代码](https://github.com/peteryipikachu-cpu/repo-name) |
```

## 4. 置顶与检查

- 在个人主页置顶 3–6 个最完整的仓库，优先可运行、文档完整、有图表或演示的项目。
- 删除或归档空仓库、重复练习与过时实验；不要把它们置顶。
- 以未登录浏览器和手机端检查：首屏 30 秒内能找到方向、项目和联系入口。
- 每季度检查外链、截图、依赖版本、README 的指标和置顶仓库是否仍准确。
